## 二、总体策略

### 2.1 推进方案

**方案A：瀑布递进**，5个阶段串行推进：

```
阶段1(地基修复) → 阶段2(前端瘦身) → 阶段3(功能补全) → 阶段4(管理增强) → 阶段5(质量收官)
   v0.9.28          v0.9.29          v0.9.30          v0.9.31          v0.9.32
```

### 2.2 选择理由

1. 风险从低到高，前面的修复为后面的功能打基础
2. 每阶段完成后可独立部署验证
3. 与已完成的4阶段代码质量优化思路一致
4. 独立开发者场景下串行最可控

### 2.3 不变性约束

- 每个阶段完成后必须通过 ruff 0错误 + mypy 0错误 + pytest全量通过
- 每个阶段完成后更新 CHANGELOG.md + VERSION.md
- 不改变现有API的公开签名（向后兼容）
- 不引入新的第三方核心依赖（除非必要）

## 四、阶段2：前端瘦身（v0.9.29）

### 4.1 目标

消除前端性能瓶颈，修复翻译遗漏，提升加载速度和用户体验。

### 4.2 任务清单

| 序号 | 任务 | 描述 | 涉及文件 |
|------|------|------|----------|
| 2.1 | 提取内联CSS | `base.html` 700行内联CSS → 合并到 `static/css/base.css` 或新建文件 | base.html, static/css/ |
| 2.2 | 提取内联JS | `index.html` 2300行内联JS → `static/js/` 新文件 | index.html, static/js/ |
| 2.3 | 修翻译遗漏 | 5处硬编码中文改用 `t()` 翻译函数，补充翻译键值 | 模板/JS文件 |
| 2.4 | CSS/JS构建 | 更新 `build.py` 处理新提取的文件 | build.py |

### 4.3 技术方案

**CSS提取**：
- 识别 `base.html` 中 `<style>` 标签内的所有CSS
- 分类：布局相关 → `base.css`，组件相关 → `components.css`，动画相关 → `animations.css`
- 移除 `<style>` 标签，确保外部CSS文件已通过 `<link>` 引入

**JS提取**：
- 识别 `index.html` 中 `<script>` 标签内的所有JS
- 按功能拆分：数据渲染 → 新文件 `index.js`，工具函数 → 已有 `utils.js`
- 保持 `data-*` 属性和模板变量的传递机制不变

**翻译修复**：
- 搜索所有模板和JS中的中文字符串
- 对比翻译键值表，找出遗漏项
- 补充翻译键值到 `translations.js` 和 `.po` 文件

### 4.4 验证标准

- [ ] 页面加载无内联CSS/JS（`<style>` 和大段 `<script>` 清零）
- [ ] 页面显示效果不变（视觉回归）
- [ ] 中英文切换无遗漏
- [ ] `build.py` 正常生成 `.min` 文件

### 4.5 风险评估

风险等级：**低**。纯前端改动，不影响后端逻辑。需注意JS提取后模板变量的传递。

## 五、阶段3：功能补全（v0.9.30）

### 5.1 目标

补齐核心功能缺口，让产品功能更完整。

### 5.2 任务清单

| 序号 | 任务 | 描述 | 涉及文件 |
|------|------|------|----------|
| 3.1 | 收藏持久化 | 前端收藏按钮 → POST API → `UserPreference` 模型存储 | 路由+JS+模板 |
| 3.2 | 新书公共API | `/api/public/new-books` + `/api/public/new-books/<publisher>` | public_api.py |
| 3.3 | 推荐公共API | `/api/public/recommendations` 端点 | public_api.py |
| 3.4 | 搜索扩展 | `SmartSearchService` 增加 NewBook 和畅销书搜索 | smart_search_service.py |
| 3.5 | 对应测试 | 每个新端点/功能补充测试用例 | tests/ |

### 5.3 技术方案

**收藏持久化**：
- 后端：`POST /api/favorites`（添加）、`DELETE /api/favorites/<book_id>`（移除）、`GET /api/favorites`（查询）
- 模型：使用已有的 `UserPreference` 模型，`preference_type='favorite'`
- 前端：`base.js` 中 `toggleFavorite()` 函数调用后端API

**新书公共API**：
- 端点：`/api/public/new-books`（所有新书）、`/api/public/new-books/<publisher>`（指定出版社）
- 响应格式：与现有 `PublicAPIResponse` 一致
- 限流：复用 `@public_rate_limit`

**推荐公共API**：
- 端点：`/api/public/recommendations`
- 参数：可选 `session_id`（用户标识）
- 逻辑：调用 `RecommendationService.get_recommendations()`

**搜索扩展**：
- `SmartSearchService.search()` 增加对 `NewBook` 模型的查询
- 增加对 `BookMetadata`（畅销书元数据）的查询
- 结果去重和排序

### 5.4 验证标准

- [ ] 收藏按钮点击后数据持久化（刷新不丢失）
- [ ] 新书/推荐公共API返回正确数据
- [ ] 搜索结果覆盖全数据类型
- [ ] 新测试全部通过

### 5.5 风险评估

风险等级：**中**。涉及新增API端点和数据库操作，需注意向后兼容和输入验证。

## 六、阶段4：管理增强（v0.9.31）

### 6.1 目标

补全运维管理能力，提供爬虫和系统监控手段。

### 6.2 任务清单

| 序号 | 任务 | 描述 | 涉及文件 |
|------|------|------|----------|
| 4.1 | 爬虫管理API | 手动触发爬虫、查看爬虫状态、查看最后运行时间 | admin.py |
| 4.2 | 系统监控 | 内存使用、数据库连接池状态、缓存命中率等指标 | admin.py |
| 4.3 | 数据备份API | 数据库导出/导入端点（管理员认证保护） | admin.py |
| 4.4 | 对应测试 | 管理端点测试用例 | tests/ |

### 6.3 技术方案

**爬虫管理**：
- `POST /api/admin/crawler/run/<publisher>`：手动触发指定出版社爬虫
- `GET /api/admin/crawler/status`：查看所有爬虫最后运行时间和状态
- 实现：调用 `publisher_crawler` 模块的 `get_all_crawlers()` 获取爬虫列表

**系统监控**：
- `GET /api/admin/system/status`：返回内存使用、数据库连接池、缓存统计
- 实现：`psutil`（已在依赖中）获取系统指标，`CacheService.get_stats()` 获取缓存指标

**数据备份**：
- `GET /api/admin/backup/export`：导出数据库为JSON
- `POST /api/admin/backup/import`：从JSON导入数据
- 安全：仅管理员可操作，操作记录日志

### 6.4 验证标准

- [ ] 爬虫可手动触发并返回状态
- [ ] 系统监控端点返回有效指标
- [ ] 数据备份/导入端点正常工作
- [ ] 所有端点需要 `@admin_required` 认证

### 6.5 风险评估

风险等级：**中**。涉及爬虫触发和数据库操作，需严格认证和日志。
